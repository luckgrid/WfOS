# Codex — metadata plane

Codex is the WfOS Level 0 **metadata-plane** product (brand). It stores the machine-readable
meaning of the system: **descriptors, registry, schemas, policies, graphs, models, and
package contracts**. See foundation doc §2.9, §2.13, and §10.

Codex has **no root folder** — it is delivered as a package inside the `wfos` workspace.
Canonical path: `Tech/Build/src/workspaces/wfos/packages/codex/` · Luckgrid disk: `Tech/Dev/...`.

## What lives here now

| Path | Kind | Purpose |
|------|------|---------|
| `descriptors/dust.descriptor.toml` | descriptor | how Dust connects (paths, CLI, modules, runtime manager) |
| `schemas/dust.tools.schema.json` | schema | contract for the generated tools registry |
| `policies/dust.agent.policy.toml` | policy | Dust agent rails (allow/block, gates) |
| `registry/tools.json` | registry | tool inventory generated by `dust doctor` (gitignored — host-specific) |
| `registry/.gitkeep` | — | keeps the registry dir tracked |

## Codex concepts (§10.1–10.7)

```txt
Descriptors  describe how things connect.
Registries   index what exists (tools, workspaces, apps, patterns, and their kinds).
Schemas      define contracts.
Policies     define rules — including agent rails and gates.
Graphs       define relationships — project deps + git resources (planned).
Models       define machine-readable domain meaning (planned).
Packages     define Hypercube-managed deliverable interfaces (planned).
```

## Relationships

- **Dust** (native substrate) produces the registry (`dust doctor`) and is governed by the
  agent policy here. Today Codex + Dust are the implemented Level 0 pair.
- **Kraken** (`krk`) and **Hypercube** (`hqb`) will read and operate on Codex metadata when
  implemented — discovery, routing, sessions, and package translation.
- **Native manifests stay authoritative** (§15.8): Codex describes meaning, routing, policy,
  and relationships; it does not replace `package.json`, `Cargo.toml`, `mise.toml`, etc.

## Interface-layer exposure (§2.9)

```txt
Toolchain layer (low)     paths, native manifests, adapter contracts, registry scans
Agent layer (mid)         descriptors, policies, scoped graphs, session context
Application layer (high)  workflow intent, domain/system labels — minimal path surface
```

## Stream metadata (§2.13)

Domain data, stream classification (`private … federated`), and privacy policy are Codex
metadata — not a root `Model/` folder. Optional domain libraries may appear under
`Tech/Dev/src/libraries/<domain>/` when filesystem expression is needed.
